2.1.1 Thinking Abstractly

  • Abstraction removes irrelevant detail so you focus on what matters.
  • Build a simple model: inputs → process → outputs.
  • Use abstraction to reduce complexity and communicate a solution clearly.

Worked example

Taxi fare model: Only keep the variables you need (distance, time, base fee). Ignore irrelevant detail (car colour, driver name). Then write a function that takes distance/time as inputs and returns the fare as the output.

Quick check

  1. Define abstraction in one sentence.
  2. Give one example of abstraction in software.
  3. Why is abstraction useful?

2.1.4 Thinking Logically

  • Use Boolean logic (AND / OR / NOT) to make decisions.
  • Conditions should be complete (cover all cases) and non-overlapping.
  • Test boundary values to prove your logic works.

Worked example

Grade boundary logic: Create ranges (e.g., ≥80, 70–79, …) then test boundary values like 79/80 to check the right branch runs.

Quick check

2.1.5 Thinking Concurrently

  • Concurrency: multiple tasks make progress in overlapping time.
  • Shared resources can cause race conditions.
  • Use locks/mutexes to protect critical sections.

Worked example

Shared counter: Two threads increment the same counter. Without a lock, updates can be lost. Put the increment inside a critical section protected by a mutex.

Quick check

  1. What is a race condition?
  2. What is deadlock?
  3. Concurrency vs parallelism: what’s the difference?

2.2.1 Programming Techniques

  • Subroutines: split a program into smaller, named parts.
  • Parameters/arguments: send data into a subroutine.
  • Return values: send a result back from a function.
  • Scope: local vs global variables.
  • Defensive programming: validate inputs and handle errors safely.

Worked example

Example: Validate an age input and return TRUE/FALSE.

FUNCTION IsValidAge(age)
  IF age >= 0 AND age <= 120 THEN
    RETURN TRUE
  ELSE
    RETURN FALSE
  ENDIF
ENDFUNCTION

  1. Define scope.
  2. Parameter vs argument: what’s the difference?

2.3.1 Analysis, Design & Comparison of Algorithms

  • Algorithms can be compared by time complexity (speed) and space complexity (memory).
  • Big-O describes how performance scales as input size (n) increases.
  • Recognise: O(1), O(log n), O(n), O(n log n), O(n²).
  • Choose algorithms based on input size, memory limits and required speed.
  • Trade-offs: faster algorithms may use more memory.

Worked example

Sorting 1,000,000 numbers:

  • Bubble sort is O(n²) → becomes too slow as n grows.
  • Merge sort is O(n log n) → scales much better for large inputs.

Conclusion: Merge sort is usually better for large datasets, even if it uses extra memory.

Quick check

  1. What does Big-O notation describe?
  2. Which grows faster for large n: n log n or n²?
  3. Give one reason an algorithm using more memory might still be chosen.

2.3.2 Algorithms for Data Structures

  • Stacks are LIFO (last in, first out) — operations: push, pop.
  • Queues are FIFO (first in, first out) — operations: enqueue, dequeue.
  • DFS explores as far as possible down a branch before backtracking (uses a stack/recursion).
  • BFS explores level-by-level (uses a queue).
  • BFS can find the shortest path in an unweighted graph.

Worked example

Exploring rooms in a building (graph):

  • DFS goes deep first, then backtracks.
  • BFS visits all neighbours first and works outward in layers.

Conclusion: Use BFS when distance from the start matters; DFS for deep exploration.

Quick check

  1. What does LIFO stand for?
  2. Which traversal algorithm uses a queue?
  3. Give one advantage of BFS over DFS.

2.3.3 Sorting Algorithms

  • You must know: bubble, insertion, merge and quick sort.
  • Bubble: repeated neighbour swaps; O(n²).
  • Insertion: builds a sorted portion; O(n²) worst-case, O(n) best-case (nearly sorted).
  • Merge: divide-and-conquer; O(n log n) but uses extra memory.
  • Quick: divide-and-conquer; O(n log n) average, O(n²) worst-case.

Worked example

Bubble sort — first pass on [7, 3, 5, 2]:

[7,3,5,2] → swap 7 & 3 → [3,7,5,2] → swap 7 & 5 → [3,5,7,2] → swap 7 & 2 → [3,5,2,7]

Merge sort (overview): split → sort halves → merge into one sorted list.

Quick check

  1. Which sort repeatedly swaps adjacent elements?
  2. Which sort uses divide and conquer?
  3. Give one advantage of insertion sort.

2.3.4 Searching Algorithms

  • Linear search checks each item one-by-one; works on unsorted data; O(n).
  • Binary search repeatedly checks the middle; requires sorted data; O(log n).
  • Binary search is faster on large datasets when the precondition (sorted) is met.

Worked example

Binary search on [3, 7, 12, 18, 21, 30, 42] for 18:

Middle is 18 → found in one comparison.

Conclusion: Binary search is efficient, but only works on sorted data.

Quick check

  1. Which search checks items one by one?
  2. What must be true before binary search can be used?
  3. Which search has time complexity O(log n)?

2.3.5 Path-Finding Algorithms

  • Graphs are made of nodes (vertices) and edges; edges may have weights (costs).
  • Dijkstra finds shortest paths using non-negative weights.
  • Dijkstra repeatedly expands the lowest-cost unvisited node.
  • A* uses a heuristic estimate to guide search toward the goal.
  • A* is often faster in practice; correctness depends on the heuristic.

Worked example

Map route planning:

  • Dijkstra guarantees the shortest path but may explore many nodes.
  • A* uses a heuristic (e.g., straight-line distance) to focus the search and often runs faster.

Conclusion: Use Dijkstra for guaranteed accuracy; A* when speed matters.

Quick check

  1. What does Dijkstra’s algorithm find?
  2. What extra feature does A* use?
  3. Why must Dijkstra’s weights be non-negative?
